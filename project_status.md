# 장바구니 물가 판단 에이전트 — 프로젝트 현황 정리

> 가천대 AI 부트캠프 · Project Day1 기준
> 작성일: 2026-07-08

---

## 0. 한 줄 요약

사용자가 "이거 지금 사도 돼?"라고 물으면, 원물·가공식품 시세를 조회하고
평년/전월 대비로 판단해 대체품이나 자취 vs 완제품 비교까지 제안하는
LangGraph 기반 에이전트.

---

## 1. 확정된 부분 ✅

### 1.1 주제 및 방향
- **주제**: 장바구니 물가 판단 에이전트 (가칭)
- **핵심 가치**: 단순 시세 조회가 아니라, "지금이 살 때인가"에 대한 **판단**과
  대체 제안까지 제공 → "API 래퍼 아니냐"는 반박에 대한 방어 논리 확보
- **7일 일정**: 가이드 마일스톤(Day1 기획~Day7 발표) 그대로 적용

### 1.2 핵심 시나리오 (2개)
1. **쌀 vs 즉석밥 비교형**: "쌀 사서 밥 짓는 거랑 햇반 사 먹는 거 뭐가 싸?"
   - 원물(쌀)과 가공식품(즉석밥)의 단위를 밥 1공기 기준으로 환산해 비교
   - 서로 다른 API를 조합 + 단위 환산이 들어가서 에이전트다움을 가장 잘 보여주는 시나리오
2. **원물 대체품 판단형**: "상추 지금 비싸? 비싸면 대체할 거 없어?"
   - 시세 조회 → 평년 대비 판정 → (비쌈 판정 시) RAG로 대체 품목 검색 → 대체품 재조회

### 1.3 기술 스택
- LLM: Upstage Solar
- Orchestration: LangGraph (StateGraph, conditional edges)
- Backend: FastAPI + SSE 스트리밍
- Vector DB: ChromaDB (RAG용)
- Infra: Docker Compose, GCP Compute Engine(GCE VM), GitHub Actions CI/CD
- LLMOps: LiteLLM(모델 게이트웨이), Auto Router(요청 복잡도별 모델 분기), Langfuse(Trace)

### 1.4 외부 API
| API | 용도 | 상태 |
|---|---|---|
| KAMIS Open-API (`dailyPriceByCategoryList`) | 원물(배추·상추·오이 등) 도소매가 조회 | ✅ 키 발급 완료, 실호출 테스트 성공 (cert_id: 5655) |
| 공공데이터포털 "가공식품 소매가격 정보" | 즉석밥·두부·김치 등 브랜드 SKU 소매가 | ❌ 미신청 (별도 발급처, data.go.kr) |

### 1.5 그래프 구조 (초안)
```
사용자 입력
   ↓
[Router] → price / knowledge / hybrid / off-topic 4분류 + 품목 추출 (구조화 출력 1회 호출)
   │
   ├─ off-topic → 거절 응답
   ├─ knowledge → RAG 검색 → 답변 생성
   ├─ price     → Tool 조회 → 판정 → 답변 생성
   └─ hybrid    → Tool 조회 → 판정 → (비쌈 판정 시) RAG 검색 → Tool 재조회 → 답변 생성 (SSE)
```

### 1.6 Tool 목록
| Tool | 역할 | 데이터 소스 |
|---|---|---|
| `parse_query` | 자연어 → 품목·수량·의도 구조화 | LLM structured output |
| `get_raw_price` | 원물 시세 조회 | KAMIS Open-API |
| `get_processed_price` | 가공식품 SKU 소매가 조회 | 공공데이터포털 (예정) |
| `normalize_unit` | 단위 환산 (생쌀↔즉석밥 등) | 내부 로직 (고정 상수) |
| `judge_price` | 평년/전월 대비 비쌈·적정 판정 | 내부 계산 |
| `search_substitute` | 대체 품목 검색 | RAG (Vector DB) |
| `search_knowledge` | 제철·보관법·가격원인 검색 | RAG (Vector DB) |
| `budget_planner` | 예산 내 장바구니 구성 | `get_raw_price` 반복 호출 |

### 1.7 RAG 문서 설계
- **콘텐츠 4종**: 제철 정보 / 대체 품목 관계 / 보관법·활용 팁 / 가격 변동 원인
- **메타데이터**: `item_name`, `category`, `season_months`, `content_type`, `source`
- **규모**: 품목 20~30개 × 콘텐츠 2~3종 = 문서 30~50개 (Day5 슬롯에 배치)

### 1.8 알려진 데이터 특성 (예외 처리 근거)
- 가격 값이 콤마 포함 문자열(`"3,606"`)로 옴 → 숫자 변환 전처리 필요
- 일부 값이 `"-"` 결측으로 옴 → 결측 처리 없으면 계산 오류
- 당일 데이터가 늦게 반영되거나 휴일엔 공백 가능 → 캐싱/Fallback 필요성의 근거

## 2. 더 디벨롭 필요한 부분 🔧

- [ ] **가공식품 API 실제 신청 및 테스트** — data.go.kr에서 "가공식품 소매가격 정보" 활용신청,
      승인 후 실호출로 스키마 확인 (KAMIS와 필드 구조가 다를 가능성 높음)
- [ ] **생쌀→밥 무게 환산 비율 확정** — 대략 2.2~2.3배로 알려져 있으나, 정확한 계수와
      "밥 1공기 = 몇 g" 기준을 팀 내에서 하나로 통일해야 계산 로직이 일관됨
- [ ] **가공식품 대표 SKU 선정 전략** — 브랜드·용량이 다양(햇반 210g×8개 등)해서
      MVP에서는 대표 SKU 1개로 고정할지, 확장 시 어떻게 넓힐지 방식 결정
- [ ] **품목코드 매핑 테이블** — 자연어 품목명("상추") → KAMIS `item_code` 매핑,
      MVP 범위(품목 20~30개)를 실제 리스트로 확정
- [ ] **Router 4분류 프롬프트/스키마 설계** — price/knowledge/hybrid/off-topic 분류를
      Pydantic 구조화 출력으로 얼마나 안정적으로 뽑을 수 있는지 검증 필요
- [ ] **Fallback/캐싱 세부 설계** — API 실패·결측 시 전일 데이터로 대체하는 로직의
      구체적인 캐싱 방식(TTL, 저장 위치) 결정
- [ ] **RAG 문서 실제 작성** — 제철정보는 KAMIS 자체 제공 자료("제철농산물 연간 50품목")
      활용 가능 여부 우선 확인, 대체품목·보관법은 팀에서 직접 정리
- [ ] **프론트엔드(Streamlit) 구현 수준** — 가이드상 평가 핵심 아니지만 시연용 최소 화면
      범위를 어디까지 할지
